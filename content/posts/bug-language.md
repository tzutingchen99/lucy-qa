# Bug 的語言

## 一個開會場景

「這個 bug 很嚴重，要馬上修。」

「等一下，你說的嚴重是指影響範圍大，還是說你覺得要排在這週？」

沉默三秒。

這不是在刁難，是因為「嚴重」這個詞在測試圈裡有兩個意思，很多人混著用。混著用的結果是：RD 以為是在說重要性，QA 說的其實是影響程度，PM 聽成是時間壓力——同一句話，三個人理解不同，會議結束了事情還沒對齊。

術語存在的原因，就是讓這種對話不用每次都重新解釋。

---

## Bug vs Defect vs Issue vs Incident

這四個詞常被當同義詞用，但不同公司、不同場合選用哪個詞其實有差。

**Bug**
最日常的說法。通常指「程式行為跟預期不符」，不分大小。口語中最常用，但也最模糊。

**Defect**
比 Bug 正式一點。強調「產品有瑕疵」，是可以量化、追蹤的東西。進 ticket 系統、寫報告的時候通常會看到這個詞。

**Issue**
更廣。不一定是程式出錯，可能是需求不清、環境問題、設計不符使用者期待。很多團隊用 Issue 包含所有「需要處理的事」，Bug 只是 Issue 的子集。

**Incident**
通常是 production 出事了。影響到真實使用者、有時間壓力、需要 post-mortem。Incident 不一定是 Bug 造成的——可能是設定錯誤、流量異常、外部依賴掛了。

| 詞 | 場合 |
|---|---|
| Bug | 日常溝通，測試中發現的問題 |
| Defect | 正式報告、追蹤系統、交付品質討論 |
| Issue | 廣義的「待處理事項」，不限於程式錯誤 |
| Incident | Production 事故，有時效性，需要應變 |

## Severity vs Priority

這組最常搞混，也最常造成排程爭議。

**Severity（嚴重度）**
這個問題的**技術影響程度**。跟多少使用者受影響、功能有多核心有關。

- Critical — 系統掛了、資料遺失、無法繞過
- High — 主要功能無法使用，沒有 workaround
- Medium — 功能受損，但可以繞過
- Low — 小問題，不影響主要流程

**Priority（優先度）**
這個問題**應該什麼時候被處理**。跟商業判斷、時程、資源有關。

Severity Low 的 bug 可以有 Priority High——例如登入頁的 logo 跑版，技術上沒什麼影響，但明天有大客戶 demo。

Severity Critical 的 bug 也可能暫時是 Priority Low——例如在幾乎沒人用的舊版功能上發現的崩潰，已確認下季要砍掉。

**Severity 是 QA 定的，Priority 是團隊一起決定的。** 把這兩件事混在一起，排程就會充滿主觀。

## Root Cause vs Symptom

大多數 bug report 只寫到 symptom。

Symptom 是你**看到**的現象：「點擊送出按鈕沒有反應。」

Root Cause 是**為什麼**：「API 在處理空字串時沒有回傳錯誤，前端 handler 收到 undefined 就靜默失敗。」

只寫 symptom 的 bug report，RD 拿到之後還要自己找 root cause。有時候找到的是錯的地方，修了一個不相關的東西，symptom 消失了，問題還在。

你不一定每次都能找到 root cause，但可以寫清楚：

- 發生了什麼（symptom）
- 什麼條件下發生（steps to reproduce）
- 你懷疑的原因（有的話寫，沒有不用硬湊）
- 不發生的條件（如果有的話）

能 reproduce，就是好 bug report 的第一條。其他都是裝飾。

## Regression Bug

修了舊問題，壞了新東西——這叫 regression。

特別麻煩的原因是它通常**不在你預期的地方出現**。你修了金流，登入掛了。你改了 API response 格式，三個下游服務靜默失敗。

找到 regression 的時候，除了修，我覺得值得問一下：**為什麼這條測試沒有在它被破壞的時候擋下來？**

通常答案是耦合太緊，或是測試的覆蓋跟真實的依賴關係脫節了。

---

## 一個結論

> **用對詞，才能讓對的人做對的事。Bug 的語言不是在咬文嚼字，是讓溝通少損耗一點。**
